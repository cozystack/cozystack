# site-router security model (Phase 1)

This note records the Phase-1 security boundary of the `site-router` app: what each guard defends, where it is enforced (guest VyOS firewall vs. Cilium pod policy vs. kube-ovn vs. the aggregated apiserver), why the guest source filter is mandatory, and the honest residuals that are deferred to tracked follow-ups. It mirrors the design's Security section and is accurate to what is implemented in this repository. The VyOS 1.5-rolling firewall syntax the guards use has been validated live against the shipped image — the `firewall ipv4 {input,forward} filter` / `firewall ipv4 name <NAME>` family and the `ipsec match-ipsec-in` / `ipsec match-none-in` matchers — so it is no longer provisional; the one residual that still needs a booted gateway is the runtime negative-security e2e proof, called out below.

The gateway is a router VM that terminates an IPsec tunnel and L3-forwards decrypted traffic onto the tenant pod network. That makes it deliberately more privileged than an ordinary tenant pod — it must source packets that are not its own pod IP, and it must be reachable from the public internet on the tunnel ports. The guards below re-contain that privilege so a compromised or misconfigured tunnel cannot become tenant-crossing, node-reaching, metadata-stealing or open internet egress.

## Threat model in one line

A remote peer (or anything that can spoof its source) sends traffic through the tunnel; the gateway must forward only traffic that is sourced from a declared remote network AND destined for the tenant's own cluster networks, must never let tunnel traffic reach the cluster's management/metadata surfaces or unrelated tenants, and must expose only the tunnel responder ports to the world while keeping its management API reachable by the platform controller alone.

## kube-ovn port_security — full relaxation + the mandatory guest source filter

OVN's per-port `port_security` drops any packet whose source IP/MAC is not the port's own — which is exactly what a router must do (it forwards packets with the remote client's preserved source IP, not the gateway pod IP). kube-ovn v1.15.10 cannot scope a CIDR in allowed-address-pairs, so Phase 1 relaxes `port_security` **fully** on the gateway VM port only (`ovn.kubernetes.io/port_security: "false"`).

**Where the relaxation is set (validated live, kube-ovn v1.15.10).** kube-ovn reconciles the logical-switch-port `port_security` field only at pod **creation** — flipping the annotation on an already-wired, live virt-launcher pod does **not** reconcile the OVN port (confirmed empirically). The relaxation is therefore **baked onto the VM pod template** (`spec.template.metadata.annotations` in `templates/vm.yaml`), so the virt-launcher pod is born with it and the OVN port is created relaxed. The controller **verifies** the pod carries the baked annotation — it no longer patches it live (a live patch would have no OVN effect) — and a finalizer's best-effort cleanup still removes it on delete.

Full relaxation removes OVN's anti-spoof for that one port, so the guest **tunnel-ingress source filter is the compensating control and is mandatory** — it is not optional hardening. Relaxing the port with no source guard would let the gateway spoof arbitrary cluster source addresses.

**The D8-ordering tradeoff (Phase-1 posture, security-relevant).** In the earlier design the controller physically held `port_security` enforcing until it had confirmed the guest source filter was live, and re-enforced it if the filter later dropped (D8: never relax before the guard is up). Because the relaxation must now be baked at pod creation, that dynamic ordering is no longer physically possible: **the port is relaxed from boot, before the controller confirms the guest source filter**. The consequences:

- The guest source filter — installed on the controller's first config push — is the **sole** compensating control. There is no OVN-level backstop while it is absent.
- There is a **boot window** between pod creation and the first successful config push during which the port is relaxed with no guest guard. This window is bounded by the fact that the tunnel is not yet up in it — there is no decrypted (spoofable) tunnel traffic to forward until the controller has pushed the tunnel config, and that same push installs the source filter.
- The controller's D8 "Ready gate" degrades from a physical control to a **status/progression** gate: it still confirms the source filter is live before treating the relaxation as effective (before reporting Ready), and if the filter later drops it re-establishes it by invalidating the cached config hash and re-pushing (the source filter, not `port_security`, is what it maintains). It no longer toggles `port_security`, because a live toggle has no OVN effect on this kube-ovn version.

Scoped `port_security` (a kube-ovn CIDR allowed-address-pair that would let the port keep anti-spoof for everything except the declared remote networks, removing the need to relax fully) remains the tracked follow-up that would close this window; see the residuals below.

The source filter lives in the guest as a VyOS named rule set (`TUNNEL-INGRESS`): it accepts established/related return traffic, then one accept per `(declared remoteCIDR × tenant-reachable cluster network)` pair, and drops everything else by default. Each accept matches **source ∈ remoteCIDR AND destination ∈ a tenant network** (the cluster pod and service CIDRs the controller reads from the `cozy-system/cozystack` ConfigMap). The destination constraint is load-bearing: a jumped-chain accept in VyOS is a terminal verdict, so a source-only accept would forward a valid-source packet to ANY destination — including the public internet — turning the gateway into unintended egress and bypassing the forward default-deny. The destination-constrained accept means a decrypted packet with a valid remote source but a non-tenant/world destination matches no accept and falls through to the default drop. With no remotes declared the rule set still emits the established/related accept and the default drop, so a resolved tunnel with an empty `remoteCIDRs` fails **closed** (drops all decrypted traffic except return), never open.

## Forward default-deny and the IPsec-match jump

The guest forward chain is default-deny (`firewall forward default-action drop`). Routed mode advertises specific remote networks and never a default route, so forwarding is denied unless a packet matches a recognised flow: established/related return traffic is accepted, **non-IPsec** traffic (locally-originated, pre-encryption tenant→remote egress) is accepted so the source filter never black-holes a tenant's own outbound flow, and **IPsec-decrypted** ingress jumps to the `TUNNEL-INGRESS` source filter above. This IPsec-match structure is the M2 redesign of an earlier design that bound the source filter to the pod-NIC inbound hook and wrongly dropped locally-originated egress; matching on the IPsec-decrypted property instead scopes the source filter to decrypted-from-tunnel traffic only.

## Two-boundary management-API isolation

The VyOS management API (HTTPS 443) must be reachable by the platform controller and by nothing else — not by the tenant, not by the remote peer over the tunnel. The appliance ships with **no SSH service and the well-known `vyos` login locked** (`encrypted-password "*"`), so there is no network shell to reach: the pod-network NIC is reachable by any same-namespace tenant workload, and a baked SSH service + the shipped `vyos`/`vyos` credential would have let any such workload SSH in and reconfigure the gateway outside the controller. The HTTPS API is therefore the only management surface, and the management firewall opens only 443 (not 22). Debug shell access is via `virtctl console` (the KubeVirt serial console, gated by cluster RBAC — never the pod network); a break-glass credential is injected out of band, none is baked into the image. Two independent boundaries enforce the API isolation because a single one has a gap.

- **Boundary A — guest local-input drop on IPsec-decrypted traffic.** A packet the gateway decrypts and addresses to itself never crosses the pod veth where Cilium enforces, so Cilium cannot see it. The guest therefore drops, as the first local-input rule, any IPsec-decrypted packet destined for ports 22/443. It matches the IPsec-decrypted property (not an interface), so it covers every tunnel interface with one rule, and it is emitted regardless of `managementCIDR` — the API must be unreachable over the tunnel even on the open-management test path.
- **Boundary B — Cilium gateway-ingress policy.** A namespaced `CiliumNetworkPolicy` selecting the gateway endpoint makes it ingress-default-deny-except-listed at the pod boundary: it admits only the public IPsec responder ports (UDP 500/4500) from the world and the controller's identity (namespace `cozy-site-router`, app `site-router-controller`) on TCP 443. The API port is never world-open.

The management firewall itself (guest local-input chain) accepts established/related return traffic, then the HTTPS API (443) only from `managementCIDR` (SSH 22 is never opened — the appliance has no SSH service), then IKE/NAT-T/ESP when a tunnel is configured and per-peer BGP (TCP 179) when BGP is enabled, and drops everything else. It is fail-closed: an empty `managementCIDR` aborts rendering unless `allowOpenManagement=true` is set explicitly (a test-only escape hatch), enforced identically in the chart (cloud-init seed) and in the controller (`--management-cidr` / `--allow-open-management`). The firewall is seeded at first boot via cloud-init so the API is protected before it is reachable, and re-stamped by the controller on every push. The `managementCIDR` value and the controller flag MUST agree (both default to the kube-ovn pod CIDR `10.244.0.0/16`); a drift silently locks the controller out of the router.

## Cilium egressDeny — metadata and management ranges

A namespaced `egressDeny` `CiliumNetworkPolicy` on the gateway endpoint denies link-local `169.254.0.0/16` (which includes the cloud metadata endpoint `169.254.169.254`) plus any operator-supplied management/node ranges (`security.egressDenyCIDRs`). Cilium deny rules take precedence over allow rules regardless of policy scope, so this namespaced deny composes correctly with the tenant's allow-only baseline (which allows egress to the world) — the deny wins. Cilium's identity model already blocks direct access to unrelated tenants, nodes and the API server; the egressDeny adds the explicit link-local + management deny that identity alone does not cover. This is the first `egressDeny` in the repository — every other cozystack policy is additive allow-only with no default-deny.

## API-key handling — not tenant-readable

The VyOS HTTPS-API token is generated by the chart (`lookup`+autogen for reconcile stability) into a per-instance Secret that is deliberately RBAC-isolated: it is NOT in the ApplicationDefinition `secrets.include` list and NOT in the tenant dashboard Role, so tenant roles cannot read it — only the controller can. It is seeded into first-boot cloud-init so the API is authenticated before it is reachable, and the VyOS API binds to the pod-network address only, with the management firewall as the compensating network control. The token (and the PSK) are redacted from every error surface: a VyOS `/configure` failure can echo the offending set-command back in its body, so the controller replaces both secrets with a placeholder before the message reaches an Event, a condition or a log.

## Deny-set validation — no cluster-blackholing remote

A tenant's declared `remoteCIDRs` must be disjoint from the cluster-owned networks, or the return route would blackhole cluster traffic. A pure `denyset.Validate` helper rejects any `remoteCIDR` that overlaps the pod / service / join / node / LB-pool networks, the always-reserved link-local (`169.254.0.0/16`) and loopback (`127.0.0.0/8`) blocks, or is a default route (`0.0.0.0/0`), or is not IPv4 (Phase 1 is IPv4-only). The same helper runs at two points so they can never disagree: a SiteRouter-scoped validating admission check rejects it synchronously at apply time (a `Forbidden` naming the offending CIDR and the colliding network), and the controller rejects it at reconcile time (a machine-readable `Ready=False`, reason `InvalidRemoteCIDR`) before it ever programs a route. Cluster CIDRs are read from the `cozy-system/cozystack` ConfigMap by both callers via one shared mapping, falling back to the platform-values defaults when a key or the ConfigMap is absent.

## Residuals (honest, deferred to follow-ups)

These are the known gaps the Phase-1 boundary accepts by decision. Each is tracked in `followups.md`.

- **Boundary B is not fully realised.** Cilium allow rules are additive, so the tenant baseline's broad allow-external/internal-communication ingress still reaches the gateway endpoint alongside the gateway-ingress policy. Fully closing it would require excluding the gateway endpoint from the tenant baseline — a shared `packages/apps/tenant` change with broad blast radius. In Phase 1 the guest VyOS firewall (Boundary A + the local-input management drop + the forward-chain guards) is the real backstop for the management API and the tunnel; the gateway CNP is the pod-boundary layer on top, not the sole defence.
- **`port_security` is fully relaxed, not scoped.** kube-ovn v1.15.10 cannot express a CIDR in allowed-address-pairs, so the gateway port loses OVN anti-spoof entirely and the guest source filter carries the load. Scoped relaxation is a tracked follow-up pending upstream kube-ovn CIDR-AAP support; the guest source filter and its negative tests stay regardless.
- **PSK is plaintext at rest.** The pre-shared key lives in a tenant-namespace Secret (base64 in etcd, not encrypted beyond whatever etcd encryption the cluster has) and is rendered into the guest VyOS configuration in plaintext. It is redacted from error surfaces, but the at-rest plaintext trade-off is inherent to a PSK-authenticated IPsec tunnel in this model.
- **The API key is seeded via cloud-init.** The token is baked into first-boot user-data so the API is authenticated before it is reachable. A controller-namespace key with post-boot rotation to a value that never appeared in cloud-init is a documented enhancement, matching the reference implementation's acknowledged trade-off.
- **Negative-security acceptance needs a runtime e2e proof.** The guard *structure* (source filter, forward default-deny, IPsec-match jump, Boundary-A drop, MSS clamp, forced encapsulation) is implemented and unit-tested, and its VyOS 1.5-rolling leaf syntax — the `firewall ipv4 {input,forward} filter` / `firewall ipv4 name <NAME>` family and the `ipsec match-ipsec-in` / `ipsec match-none-in` matchers — is validated live against the shipped image (a full render-equivalent config commits cleanly). What is still deferred is the runtime proof on a booted two-VM topology that an undeclared-source packet, and a valid-source/world-destination packet, are both actually dropped; that is the e2e negative-security suite, blocked on the published golden image (see `followups.md`).
- **Deny-set node/LB-pool discovery is best-effort.** Node subnets and LB pools are not cleanly discoverable as cluster facts, so the deny-set enforces against pod/service/join plus the always-reserved networks and skips node/LB-pool unless supplied. A default pod-CIDR cluster is covered; a custom `networking.podCIDR` needs `managementCIDR` (and the controller flag) set to match.
