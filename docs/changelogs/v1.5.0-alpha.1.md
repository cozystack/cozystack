# Cozystack v1.5.0-alpha.1 — "TLS-Native Services & Gateway Evolution"

This alpha release advances the platform along several major axes: every stateful managed service now sports opt-in TLS with operator- or cert-manager-managed certificates; the Cilium-backed Gateway API becomes the new default ingress layer for Cozystack-native services; dynamic dropdown support lands in dashboard forms through the new `Option` resource; GPU passthrough works out-of-the-box for tenant Kubernetes clusters; and the backup subsystem adds both a generic Job strategy and an etcd strategy together with a platform-managed default BackupClass.

> **⚠️ Alpha Release Warning**: This is a pre-release version intended for testing and early adoption. Breaking changes may occur before the stable v1.5.0 release.

---

## Feature Highlights

### Gateway API Support via Cilium

Cilium Gateway API replaces ingress-nginx as the default ingress layer for Cozystack-native services. The platform renders one `TenantGateway` CR per tenant that owns its own Gateway, reconciled by `cozystack-controller` into actual `Gateway`, per-tenant `Issuer`, `Certificate`, and HTTP→HTTPS redirect `HTTPRoute` resources. Tenants that do not explicitly set `tenant.spec.gateway=true` inherit the Gateway of the nearest ancestor that owns one — the same inheritance model used by the existing ingress layer — so the upgrade is transparent for the majority of tenants.

Two TLS modes are supported: HTTP-01 (default, per-app certificates, zero platform configuration) and DNS-01 (opt-in, single wildcard certificate per apex, plus per-child SAN expansion for inheriting tenants). Four DNS-01 providers are wired in: Cloudflare, Route53, DigitalOcean, and RFC 2136. In both modes the per-tenant `LoadBalancer` Service draws its external IP from the cluster's existing LB allocator (MetalLB, Cilium LB-IPAM, or any other) without tenant-chart-level coupling.

The breaking change that accompanies this feature: the `publishing.exposure=loadBalancer` mode introduced in a prior release is removed because it caused traffic collisions in the default Cozystack configuration. Operators who explicitly opted into that mode will have their ingress Service revert to `ClusterIP + spec.externalIPs` on upgrade; the default `externalIPs` path is unaffected.

Learn more in the [Cilium Gateway API documentation](https://cozystack.io/docs/networking/gateway-api).

### TLS Support for Managed Services

Four managed services — Kafka, NATS, Qdrant, and Postgres — gain opt-in TLS with a unified tri-state `tls.enabled` field: `null` (inherit from `external`; TLS is on when the service is published externally), `true` (force TLS on), `false` (force TLS off). Explicit values always win over the inherited default.

Each service uses its natural PKI model: Kafka delegates entirely to the Strimzi operator, which publishes cluster-CA and client-CA secrets; Postgres uses CNPG operator-managed certificates with external-hostname injection; NATS and Qdrant render self-contained cert-manager chains (self-signed Issuer → CA Certificate → CA Issuer → leaf Certificate) in the tenant namespace, with proper SAN sets covering internal DNS forms plus the external hostname when `external: true`.

All four integrations have been verified end-to-end on sandbox clusters. Trust anchors for clients are documented per-service.

### Dynamic Form Dropdowns via `Option` Resource

A new `core.cozystack.io/v1alpha1 Option` resource enables dashboard form fields to render as dropdowns populated from live cluster resources. Fields such as GPU device names, KubeVirt instancetypes/preferences, Multus networks, VM images, storage pools/classes, and backup classes/plans were previously free-text or static enums that drifted from reality; they now pull from in-process providers backed by a privileged client so tenants see curated, up-to-date lists without requiring broad cluster read permissions.

App `values.yaml` files declare the binding via the new `@x-cozystack-options` schema directive (e.g. `## @x-cozystack-options {source: instancetype}`), which is preserved through code generation into the served OpenAPI so the dashboard widget can discover the provider from the schema alone. Providers cover: `gpu`, `instancetype`, `instanceprofile`, `network`, `image`, `storagepool`, `storageclass`, `backupclass`, `plan`, `backup`, and `appkind`.

### GPU Passthrough Out-of-the-Box

Tenant Kubernetes clusters using GPU passthrough no longer require manual post-install fixups. Two defaults are now applied automatically: GPU node-groups are labeled `gpu=on` at kubelet registration (so HAMi's `hami-device-plugin` DaemonSet schedules and starts advertising `nvidia.com/gpu`), and the tenant gpu-operator loads the NVIDIA driver with `NVreg_NvLinkDisable=1` (so single-SXM-GPU passthrough — where the VM has no NVSwitch — no longer hangs on `Fabric State: In Progress` / `system not yet initialized`). Both defaults remain overridable via `addons.gpuOperator.valuesOverride`.

### Backup Subsystem Improvements

Three complementary backup improvements land together in this release:

- **Job backup strategy** (#1721): a generic, application-agnostic driver that runs a user-supplied `batch/v1.Job` template for both backup and restore, using the same parameter round-trip and lifecycle conventions as the existing Velero and Altinity strategies. The template context exposes `.Application`, `.Release.Name/.Namespace`, `.Mode` (`backup`/`restore`), `.Parameters`, and `.Backup` (restore only).
- **etcd backup strategy** (#2641): a cluster-scoped S3-backed strategy for etcd snapshots, with destructive in-place restore support and snapshot metadata persistence.
- **Platform default BackupClass** (#2716): the platform now ships a `cozy-default` BackupClass, a `cozy-backups` system bucket, and an opt-in `useSystemBucket` flag, so operators can enable backup for any app without configuring per-release credentials.

### Deletion-Protection Guardrail

A `ValidatingAdmissionPolicy` is now installed on Kubernetes 1.30+ clusters that blocks `DELETE` on critical platform objects labeled `platform.cozystack.io/no-delete=true`. Protected resources include `cozy-system` and `tenant-root` Namespaces, the `tenant-root` HelmRelease, the `cozystack-version` ConfigMap, `cozystack-packages` OCIRepository, cluster-wide cert-manager Issuers, LinstorCluster, and the CRDs for packages/packagesources. The VAP and its Binding are themselves protected. No webhook, DaemonSet, TLS certificate, or extra image is required — evaluation is in-process CEL.

To bypass (e.g. for legitimate teardown): `kubectl label <kind> <name> platform.cozystack.io/no-delete-`, then delete.

## Major Features and Improvements

* **feat(networking): Gateway API support via Cilium**: Replaces ingress-nginx with a controller-reconciled Cilium `Gateway` per tenant, supporting HTTP-01 and DNS-01 cert modes, per-tenant or inherited Gateway attachment, and DNS-01 SAN expansion for child tenants ([**@lexfrei**](https://github.com/lexfrei) in #2470).

* **feat(api): add Option resource and x-cozystack-options for dynamic form dropdowns**: Introduces the namespaced `core.cozystack.io/v1alpha1 Option` read-only resource with 11 in-process providers and the `x-cozystack-options` schema keyword, enabling dashboard fields to render as live-populated dropdowns ([**@kvaps**](https://github.com/kvaps) in #2778).

* **feat(kubernetes): enable GPU passthrough out-of-the-box (NvLinkDisable + gpu=on)**: Automatic `gpu=on` node labeling and `NVreg_NvLinkDisable=1` driver default for tenant GPU clusters, making single-SXM-GPU passthrough work without manual intervention ([**@kvaps**](https://github.com/kvaps) in #2780).

* **feat(rbac): allow tenants to start/stop/restart their VMs**: Adds `virtualmachines/{start,stop,restart}` subresource permissions to `cozy:tenant:use:base`, resolving 403 errors on dashboard VM power controls for all tenant roles ([**@kvaps**](https://github.com/kvaps) in #2777).

* **feat(backups): implement Job backup strategy**: Generic `strategy.backups.cozystack.io/Job` driver that runs a user-supplied Kubernetes Job for application-agnostic backup and restore operations ([**@lllamnyp**](https://github.com/lllamnyp) in #1721).

* **feat(platform): add implementation of backup-strategy controller for etcd**: Cluster-scoped S3-backed etcd snapshot strategy with destructive in-place restore and snapshot metadata persistence ([**@androndo**](https://github.com/androndo) in #2641).

* **feat(platform): add default backupclass**: Platform-managed `cozy-default` BackupClass and `cozy-backups` system bucket with opt-in `useSystemBucket` flag, enabling zero-configuration backup for any application ([**@androndo**](https://github.com/androndo) in #2716).

* **feat(platform): add deletion-protection guardrail via ValidatingAdmissionPolicy**: In-process CEL VAP that blocks DELETE on platform objects labeled `platform.cozystack.io/no-delete=true`, protecting critical cluster resources on Kubernetes 1.30+ ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2650).

* **feat(kafka): add TLS support via Strimzi listener configuration**: Tri-state `tls.enabled` field controlling the external Kafka listener (port 9094), with the Strimzi operator owning the full cluster PKI ([**@Arsolitt**](https://github.com/Arsolitt) in #2681).

* **feat(nats): add TLS support via cert-manager**: Tri-state `tls.enabled` field with a self-contained cert-manager chain (self-signed CA → leaf cert) covering internal DNS forms and the external hostname ([**@Arsolitt**](https://github.com/Arsolitt) in #2684).

* **feat(qdrant): add TLS support via cert-manager**: Tri-state `tls.enabled` field with a self-contained cert-manager chain covering REST and gRPC endpoints, plus per-replica short-form SANs for p2p TLS hostname verification ([**@Arsolitt**](https://github.com/Arsolitt) in #2685).

* **feat(postgres): add TLS support via CNPG operator-managed certificates**: Tri-state `tls.enabled` with CNPG operator-managed PKI; when `external: true`, the chart injects the external hostname into the operator's SAN list ([**@Arsolitt**](https://github.com/Arsolitt) in #2686).

* **feat(platform): add kuberture as optional system package**: Adds [kuberture](https://github.com/lexfrei/kuberture) as an opt-in system package that translates the `default/kubernetes` EndpointSlice into annotated headless Services for external-dns consumption, enabling DNS publication of the Kubernetes API endpoint ([**@lexfrei**](https://github.com/lexfrei) in #2647).

* **feat(apps): mark stateful-app storageClass fields as immutable**: Annotates `storageClass` as immutable via `x-kubernetes-validations` across all stateful charts (clickhouse, foundationdb, harbor, http-cache, kafka, kubernetes, mariadb, mongodb, nats, openbao, opensearch, postgres, qdrant, rabbitmq, redis, vm-disk), preventing the common mistake of editing a field Kubernetes fixes at PVC creation time ([**@lexfrei**](https://github.com/lexfrei) in #2639).

* **feat(flux): upgrade to v2.8.0 + chart fixes for strict SSA & kstatus**: Upgrades flux-operator/flux-instance to v0.48.0/v0.50.0 and Flux to v2.8.0, fixing misplaced fields that strict SSA now rejects, removing deprecated `upgrade.force: true`, and resolving a circular wait deadlock under the new kstatus health checking ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2602).

* **refactor(seaweedfs): split seaweedfs-system into seaweedfs-db + seaweedfs-system**: The CNPG `Cluster/seaweedfs-db` moves to its own HelmRelease with `healthCheckExprs`-based readiness gating (Flux v2.8 feature), eliminating the `seaweedfs-filer` CrashLoopBackOff race on fresh tenant installs ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2601).

* **feat(dashboard): add cluster-usage RBAC for the new admin page**: Grants the dashboard service account the RBAC permissions required for the new cluster-usage admin page ([**@lexfrei**](https://github.com/lexfrei) in #2743).

## Fixes

* **fix(platform): add OpenSearch to PaaS bundle**: The two missing `cozystack.opensearch-*` PackageSources are now included in the PaaS bundle template, allowing tenants to provision OpenSearch instances ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2648).

* **fix(csi): route RWX Block volumes to upstream hotplug detach**: Replaces the access-mode-only `hasRWXAccessMode` predicate with `isNFSVolume` (RWX + Filesystem mode), preventing live-migration disks from being silently skipped by the NFS cleanup branch and leaving stale `DataVolume` entries ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2658).

* **fix(csi): verify VMI Ready after kubevirt-csi Publish to surface stuck-PVC failures**: Re-reads `VMI.Status.VolumeStatus` after upstream `ControllerPublishVolume` returns success and returns `codes.Unavailable` if the volume is not in `VolumeReady` phase, preventing silent false-success for stuck-PVC conditions ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2659).

* **fix(platform): fix broken dashboard login by token**: Restores token-based dashboard login which was broken by a recent platform change ([**@androndo**](https://github.com/androndo) in #2661).

* **fix(kafka): remove ZooKeeper PVCs on uninstall**: Kafka release deletion now properly reclaims ZooKeeper PVCs, preventing orphaned storage after uninstall ([**@Arsolitt**](https://github.com/Arsolitt) in #2679).

* **fix(postgres): accept integer values for postgresql.parameters in schema**: The `postgresql.parameters` map now accepts integer values in addition to strings, matching the CNPG API and fixing validation failures for parameters such as `max_connections` ([**@IvanHunters**](https://github.com/IvanHunters) in #2687).

* **fix(platform): migrate ephemeralStorage to diskSize via pre-upgrade hook**: Adds a pre-upgrade hook migration (43) that renames the `ephemeralStorage` field to `diskSize` for affected resources, preventing upgrade failures on clusters with the old field name ([**@IvanHunters**](https://github.com/IvanHunters) in #2688).

* **fix(opensearch-operator): replace deprecated kube-rbac-proxy image**: Replaces the deprecated `kube-rbac-proxy` sidecar image in the opensearch-operator deployment with the current upstream image ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2689).

* **fix(info): use root-host for Keycloak OIDC issuer URL in tenant kubeconfig**: Corrects the Keycloak OIDC issuer URL written into tenant kubeconfig files, preventing OIDC authentication failures for tenant cluster users ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2704).

* **fix(cluster-api): fix kamaji OOM and set limits on unset providers**: Adds memory limits to kamaji and unset cluster-api provider deployments, preventing OOM kills that silently break tenant cluster provisioning ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2708).

* **fix(kubernetes): add config_path patch for containerd 2.x**: Adds a sed patch for the `plugins."io.containerd.grpc.v1.cri"` → `plugins."io.containerd.cri.v1.images"` config path change in containerd 2.x, allowing worker nodes to join on clusters running the newer containerd ([**@elaugaste**](https://github.com/elaugaste) in #2723).

* **fix: startupProbes for kubevirt-cdi-operator and grafana-operator**: Adds `startupProbe` to both operators so slow startup (slow leader-election for kubevirt-cdi-operator, high load for grafana-operator) no longer triggers premature liveness kills ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2725).

* **fix(platform): add cert-manager dependency to webhook-cert consumers**: Adds explicit `dependsOn: cert-manager` to HelmReleases that mount webhook certificates issued by cert-manager, preventing races where webhook endpoints come up before the Issuer is ready ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2726).

* **fix(dashboard): unblock token-proxy startup when JWKS is briefly unreachable**: The token-proxy no longer fails permanently when the JWKS endpoint is temporarily unreachable during startup; it retries, so transient OIDC provider hiccups no longer leave the dashboard login broken ([**@lexfrei**](https://github.com/lexfrei) in #2745).

* **fix(kubernetes): stamp application lineage labels on worker node VMs**: Worker node VirtualMachine objects now carry the same `application.cozystack.io/*` lineage labels as other Cozystack-managed resources, enabling consistent filtering and display in the dashboard ([**@kvaps**](https://github.com/kvaps) in #2779).

## Breaking Changes & Upgrade Notes

* **revert(ingress)!: remove broken publishing.exposure=loadBalancer mode**: The `publishing.exposure=loadBalancer` mode (introduced in #2468) is removed. It caused two failure modes in default Cozystack: IP collision with the existing ingress-nginx DNAT, and an unannounced Cilium LB pool. Operators who explicitly set `publishing.exposure=loadBalancer` will have their ingress Service revert to `ClusterIP + spec.externalIPs` on upgrade. Default installations using `externalIPs` are unaffected. A mechanism-agnostic replacement will be introduced in a follow-up ([**@kvaps**](https://github.com/kvaps) in #2603).
