# Cozystack v1.5.0

Cozystack v1.5.0 ships **Flux shard-operator for tenant Helm-controller sharding**, **Cilium-backed Gateway API** as a first-class networking layer, **TLS support for all major managed services** (Kafka, NATS, Qdrant, PostgreSQL), an **operator-provided wildcard certificate** for platform ingress, an **Option resource for dynamic dashboard dropdowns**, full **HelmRelease generation knobs** surfaced as operator and API flags, **tenant VM start/stop/restart RBAC**, **GPU passthrough auto-wiring**, a new **Job backup strategy**, an **etcd backup-strategy controller**, **velero installed by default**, and a **tenant overview Grafana dashboard** for platform admins. The release rolls up every fix from v1.4.1 through v1.4.4.

Platform component bumps: **Flux v2.7.3 → v2.8.0**, **MetalLB v0.15.2 → v0.16.1**, **SeaweedFS 4.05 → 4.31**, **ouroboros v0.7.2 → v0.8.0**, **flux-operator v0.48.0**, and **seaweedfs-cosi-driver v0.3.1** *(backported to v1.4.3)*.

> **Note:** Items marked *(backported to v1.4.x)* were also shipped in v1.4.1, v1.4.2, v1.4.3, or v1.4.4 patch releases.

## Feature Highlights

### Flux Shard Operator: Tenant Helm-Controller Sharding

The new `flux-shard-operator` system package spreads tenant HelmReleases across multiple dedicated helm-controller shards, preventing a noisy tenant — such as one with a HelmRelease stuck in infinite remediation — from starving the reconciliation queue for other tenants.

The operator has two parts: a **placement controller** that assigns each tenant to a shard using greedy least-loaded distribution weighted by HelmRelease count, and a **shard runtime** that provisions the correct number of helm-controller Deployments cloned from the existing `flux-aio` Deployment — so images and feature gates remain version-synced automatically. The unit of placement is the tenant namespace (all of its HelmReleases share one shard label), and the controller supports pinned tenants, threshold-gated rebalance, paced moves, and never moves tenants that are being deleted. Watches are metadata-only, so the placement controller does not decode the helm-controller status-patch firehose.

`flux-aio` itself is unchanged and continues to serve all tenants without a shard key in clusters that choose not to enable the operator ([**@kvaps**](https://github.com/kvaps) in #2821).

### Gateway API Support via Cilium

Cozystack now ships **Cilium Gateway API** as a first-class networking layer alongside ingress-nginx. A new `TenantGateway` CRD is reconciled by `cozystack-controller`, which provisions a `Gateway`, per-tenant `Issuer`, `Certificate` (wildcard or per-listener), and HTTP→HTTPS redirect `HTTPRoute` from a single tenant-level declaration. The Helm chart does not render Gateway or Certificate resources directly, preventing the Helm-vs-controller race on `Gateway.spec.listeners` that route-driven listener additions would otherwise trigger.

Tenants opt in by setting `spec.gateway` on their Tenant CR. A child tenant inherits the parent's Gateway by default via the `namespace.cozystack.io/gateway` label. DNS-01 SAN expansion lets child tenants register their apex domains as additional SANs on the parent wildcard certificate without a separate ACME round trip. The platform toggles ingress-nginx ↔ HTTPRoute automatically based on `_cluster.gateway-enabled`. External-DNS gains `gateway-tlsroute` support alongside `gateway-httproute`.

All existing ingress-nginx flows are unaffected — the Gateway API layer is purely additive and disabled by default ([**@lexfrei**](https://github.com/lexfrei) in #2470).

### TLS for Managed Services: Kafka, NATS, Qdrant, and PostgreSQL

All four major stateful managed services now support TLS, using a consistent tri-state `tls.enabled` field:

- **null / unset** — auto-inherits the value of `external` (TLS on when the service is published externally, off for cluster-internal deployments). This keeps the default behaviour unchanged for existing deployments.
- **true** — TLS always on.
- **false** — TLS always off.

**Kafka** wires TLS through the Strimzi listener configuration. The operator manages the cluster PKI end-to-end and publishes `<release>-cluster-ca-cert` and `<release>-clients-ca-cert` for client trust. The external LoadBalancer listener on port 9094 is independently gated on `external: true`; `tls.enabled` only governs whether that listener uses TLS ([**@Arsolitt**](https://github.com/Arsolitt) in #2681).

**NATS** and **Qdrant** use a self-contained cert-manager chain rendered in the tenant namespace: a self-signed Issuer, CA Certificate, CA Issuer, and leaf Certificate with SAN coverage for all three DNS forms per service plus the external hostname when `external: true`. Qdrant adds short-form per-replica SANs for p2p TLS hostname verification and a Reloader annotation to trigger pod restarts on cert rotation ([**@Arsolitt**](https://github.com/Arsolitt) in #2684 and #2685).

**PostgreSQL** uses the CNPG operator's built-in cert management instead of a chart-rendered chain. The operator auto-generates a self-signed CA and signs server, client, and replication certs. When `external: true` and TLS is on, the chart sets `spec.certificates.serverAltDNSNames` so the operator appends the external hostname to the leaf cert. The trust anchor for clients is the `<release>-ca` Opaque Secret (key `ca.crt`) ([**@Arsolitt**](https://github.com/Arsolitt) in #2686).

### Operator-Provided Wildcard Certificate for Platform Ingress

Operators can now serve platform services and the root tenant's ingress under a pre-existing wildcard TLS certificate by setting a single `publishing.certificates.wildcardSecretName` value that names a TLS Secret they have already provisioned. Only the Secret name travels through the `cozystack-values` channel — the certificate and private key are never broadcast to tenant namespaces.

When a wildcard Secret is configured, `ingress-nginx` is reconfigured to use it as the default SSL certificate, eliminating the need for per-host ACME certificates on the root domain. Existing per-host ACME flows are unaffected for operators who do not set this value ([**@lexfrei**](https://github.com/lexfrei) in #2819).

### Option Resource and Dynamic Dashboard Dropdowns

A new `core.cozystack.io/v1alpha1 Option` resource provides **runtime-populated dropdowns** for the dashboard's create and edit forms. Fields that reference live cluster resources — GPU device names, KubeVirt instancetypes and preferences, Multus networks, VM images, bucket storage pools, storage classes, backup classes and plans — can now render as curated option lists instead of free-text inputs that drift from reality.

Option items are computed on read by an in-process provider registry using a privileged client, so tenants see current values without needing broad cluster read access. Providers cover: `gpu`, `instancetype`, `instanceprofile`, `network`, `image`, `storagepool`, `storageclass`, `backupclass`, `plan`, `backup`, and `appkind`.

Chart authors expose the feature via a new `@x-cozystack-options` directive in `values.yaml`; `cozyvalues-gen` propagates it through all generated artifacts (schema, Go types, and `-rd` ApplicationDefinitions) so the extension is carried into the served OpenAPI ([**@kvaps**](https://github.com/kvaps) in #2778).

### HelmRelease Generation Knobs and Per-App Timeouts

The cozystack API server now exposes the same HelmRelease generation flags introduced for the operator in v1.4.0: `--helmrelease-interval`, `--helmrelease-retry-interval`, `--helmrelease-install-timeout`, `--helmrelease-upgrade-timeout`, and `--helmrelease-max-history`. Previously, the `cozystack-api` path that converts Application CRs to HelmReleases hardcoded its own values (5-minute interval, `Remediation{Retries:-1}`) while the operator path used flags — creating divergent behaviour for the same application kind depending on which code path generated the HelmRelease.

A companion change introduces a **per-Application Install/Upgrade timeout** driven by a `release.cozystack.io/helm-upgrade-timeout` annotation on ApplicationDefinition. This extends the existing `helm-install-timeout` annotation support so individual application kinds can declare asymmetric install vs. upgrade budgets — for example a managed service with a fast install but a slow rolling upgrade ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2571).

## Major Features and Improvements

* **feat(rbac): allow tenants to start, stop, and restart their VMs**: Grants tenant users permission to call the KubeVirt VM power-control subresources (`virtualmachines/{name}/start|stop|restart`) by adding them to the `cozy:tenant:use:base` ClusterRole. Previously the dashboard VM power buttons returned 403 even for tenant admins, while VNC and console subresources were already granted at the same level ([**@kvaps**](https://github.com/kvaps) in #2777).

* **feat(kubernetes): enable GPU passthrough out-of-the-box**: Tenant Kubernetes clusters with GPU node-groups now have passthrough working without manual intervention. Worker nodes in GPU node-groups are automatically labeled `gpu=on`, and the tenant gpu-operator is configured to load the NVIDIA driver with `NVreg_NvLinkDisable=1` to avoid NvLink errors ([**@kvaps**](https://github.com/kvaps) in #2780).

* **feat(platform): auto-wire KubeVirt `permittedHostDevices` and `HostDevices` feature gate when `gpu-operator` is enabled**: When the passthrough or vGPU variant of `cozystack.gpu-operator` is deployed, the platform now automatically patches the KubeVirt CR to add `HostDevices` to `developerConfiguration.featureGates` and list the detected GPU resources under `permittedHostDevices`. Previously the GPU Operator reported green and the resource appeared in node allocatable, but every VM referencing it via `domain.devices.hostDevices` was rejected by the admission webhook ([**@lexfrei**](https://github.com/lexfrei) in #2768).

* **feat(gpu-operator): add container variant for hosts with a preinstalled driver**: A third `cozystack.gpu-operator` variant is now available for hosts where the NVIDIA driver and `nvidia-container-toolkit` are installed by the OS package manager (the typical Ubuntu/Debian GPU node shape). The existing `passthrough` variant runs `vfioManager` to bind `vfio-pci` — which conflicts with an active host driver — and `vgpu` requires a proprietary vGPU Manager; neither worked on this host shape ([**@lexfrei**](https://github.com/lexfrei) in #2766).

* **feat(platform): add deletion-protection guardrail via `ValidatingAdmissionPolicy`**: A new VAP blocks `DELETE` on any platform object labeled `platform.cozystack.io/no-delete=true` using native in-process CEL evaluation — no webhook DaemonSet, no Service, no TLS, no additional image. The policy and its binding are themselves self-protected, and a `migration 42` backfills the label on the `cozystack-version` ConfigMap. The policy is gated on Kubernetes 1.30+ capability detection ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2650).

* **feat(backups): implement Job backup strategy**: Adds a generic, application-agnostic `strategy.backups.cozystack.io/Job` driver that runs a user-supplied Kubernetes `Job` to create and restore backups. This complements the application-specific strategy controllers and allows operators to wire any workload into the backup framework ([**@lllamnyp**](https://github.com/lllamnyp) in #1721).

* **feat(platform): add backup-strategy controller for etcd**: Extends the backupstrategy-controller with an etcd-specific driver, allowing operators to declare backup strategies for etcd clusters managed by etcd-operator. The controller composes `EtcdBackup` and `EtcdBackupSchedule` CRs and reconciles them against an S3-compatible object store ([**@androndo**](https://github.com/androndo) in #2641).

* **feat(platform): add default backupclass**: The platform now ships a default `BackupClass` resource so backup strategies work out-of-the-box without operators needing to create a class manually on fresh installs ([**@androndo**](https://github.com/androndo) in #2716).

* **feat(platform): install velero by default**: The `backupstrategy-controller` declares a hard dependency on `cozystack.velero`, but Velero remained an optional package — causing the controller's Package to stay `DependenciesNotReady` indefinitely on any bundle that did not explicitly enable it. Velero is now enabled by default in the platform bundle ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2833).

* **feat(apps): mark `storageClass` fields as immutable on stateful applications**: The `storageClass` field on stateful app types (including managed databases and Kubernetes worker node-groups) is now marked immutable via the `x-kubernetes-immutable` schema extension, preventing silent in-place changes that would require manual PVC migration ([**@lexfrei**](https://github.com/lexfrei) in #2639).

* **feat(platform): add kuberture as optional system package**: [kuberture](https://github.com/leaseweb/kuberture) is now available as an opt-in system package (`bundles.enabledPackages`). kuberture publishes the managed Kubernetes API endpoint as a DNS record via external-dns, enabling `kubectl` access to tenant clusters without a LoadBalancer or manual DNS entry ([**@lexfrei**](https://github.com/lexfrei) in #2647).

* **feat(monitoring): add tenant overview dashboard**: A new **Tenant Overview** dashboard is available in the root/infra Grafana. It shows fleet-wide totals (tenants, apps, pods, CPU/memory/storage) plus a per-tenant leaderboard with resource utilization, app counts, and not-ready apps, and Top-N bar gauges by CPU, memory, storage, and app count — giving platform admins a single view of which tenants consume what ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2809).

* **feat(dashboard): add cluster-usage RBAC for the new admin page**: Adds `get/list/watch` on `namespaces` and the cluster-usage aggregation endpoint to the `cozy:infra:admin` ClusterRole so the new admin resource-usage page in the dashboard renders without 403 errors ([**@lexfrei**](https://github.com/lexfrei) in #2743).

* **feat(flux): upgrade to Flux v2.8.0**: Upgrades Flux from v2.7.3 to v2.8.0 across the vendored tenant chart and the embedded management-cluster manifests. Flux v2.8's helm-controller v1.5.0 introduces **Server-Side Apply with `--force-conflicts`**, enabling strict CRD schema enforcement — fields that were silently dropped on v2.7 now fail the apply. A set of chart fixes are bundled to address fields that became hard errors under the new SSA regime ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2602).

* **refactor(seaweedfs): split `seaweedfs-system` into `seaweedfs-db` + `seaweedfs-system`**: The CNPG `Cluster/seaweedfs-db` resource is moved to its own HelmRelease (`seaweedfs-db`) that `seaweedfs-system` depends on. The new HR uses Flux v2 `HelmRelease.spec.healthCheckExprs` with a CEL expression on `Cluster.status.conditions[type=Ready]`, so `seaweedfs-system` reaches `Ready=True` only after the PostgreSQL primary is actually serving connections. This eliminates the `seaweedfs-filer` CrashLoopBackOff race during fresh tenant installs where Cilium's socket-LB returns `EPERM` on connections to ClusterIPs with no ready endpoints ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2601).

* **feat(ouroboros): bump to v0.8.0** *(backported to v1.4.3)*: Upgrades ouroboros to v0.8.0, which logs an explicit reason when its TCP backend readiness check fails — making stuck-proxy situations immediately diagnosable. Also migrates the kubectl sidecar image from Docker Hub to `mirror.gcr.io`, avoiding anonymous pull rate-limits on shared runner IPs ([**@lexfrei**](https://github.com/lexfrei) in #2807).

* **chore(metallb): bump MetalLB to v0.16.1**: Upgrades the vendored MetalLB chart from v0.15.2 to v0.16.1 and rebuilds the `metallb-controller` and `metallb-speaker` images. Adopts the upstream-default FRR-K8s BGP backend (FRR mode is deprecated in v0.16). The cozystack-built images move in lockstep with the chart to preserve the new probe contract ([**@lexfrei**](https://github.com/lexfrei) in #2699).

* **chore(seaweedfs): bump SeaweedFS to 4.31**: Upgrades the vendored SeaweedFS chart and images to version 4.31 ([**@lexfrei**](https://github.com/lexfrei) in #2834).

* **feat(hack): rewrite `check-readiness` as a Go command with test suite**: Replaces the `hack/check-readiness.sh` diagnostic with a Go command at `cmd/check-readiness`, integrated into the build the same way as `cozypkg`. The rewrite adds a test suite and makes the tool composable as a library for future automation ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2755).

* **chore(maintainers): mark @lllamnyp as Core Maintainer**: Aleksei Sviridkin ([**@lllamnyp**](https://github.com/lllamnyp)) is promoted to Core Maintainer, reflecting sustained contributions across API design, CSI, and testing infrastructure ([**@tym83**](https://github.com/tym83) in #2719).

## Bug Fixes

* **fix(csi): detach orphan hot-plug volumes when VMI outlives its VM**: During node reprovisioning in tenant Kubernetes clusters, the VMI and its hot-plug pod can outlive the parent VirtualMachine. The upstream driver treated a missing VM as "detach succeeded", leaving the hot-plug pod alive and the infra storage device exclusively attached to the source host — every subsequent attach of that volume failed with `failed to set source device readwrite`. The wrapper now re-checks `VMI.status.volumeStatus` after upstream returns success and, if the hot-plug is still reported, detaches it from the VMI directly via the `removevolume` subresource ([**@kvaps**](https://github.com/kvaps) in #2866).

* **fix(api): publish open spec as `x-kubernetes-preserve-unknown-fields`, not `additionalProperties:true`**: The aggregated `apps.cozystack.io` resources published a boolean `additionalProperties: true` for their free-form `.spec`, which caused `kube-controller-manager`'s ValidatingAdmissionPolicy type-checker to nil-dereference when it processed any VAP matching those resources. This triggered a `CrashLoopBackOff` on every control-plane node and stalled reconciliation. Switching to `x-kubernetes-preserve-unknown-fields: true` is semantically equivalent but handled safely by the type-checker ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2867).

* **fix(platform): order `cozystack-basics` after the APIs its admission policies type-check**: `cozystack-basics` ships cluster-wide ValidatingAdmissionPolicies that match `tenants.apps.cozystack.io` and Gateway API types. On a cold install, if these policies were created before `cozystack-api` was Ready, `kube-controller-manager` would nil-dereference while type-checking the policy against the not-yet-served `tenants` schema, enter `CrashLoopBackOff`, stall cert and token issuance, and prevent `cozystack-api` from ever starting — a deadlock. `cozystack-basics` now `dependsOn` both `cozystack-api` and the Gateway API CRD installer ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2842).

* **fix(seaweedfs): restore S3 TLS and COSI provisioner SA after SeaweedFS 4.31 upgrade**: The 4.31 chart bump nested the `-cert.file`/`-key.file` TLS args inside `{{- if .Values.s3.httpsPort }}`, but Cozystack ships `httpsPort: 0` — so no TLS args rendered, S3 spoke plaintext HTTP while all probes and clients expected HTTPS. Additionally, the COSI provisioner Deployment referenced a ServiceAccount by a different name than what the chart creates, causing pod startup failures. Both regressions are fixed via vendored patches ([**@kvaps**](https://github.com/kvaps) in #2916).

* **fix(seaweedfs): restore `-lock` BucketClass, S3 service name, and drop `volumeSizeLimitMB`**: Three additional regressions from the SeaweedFS 4.31 bump: the `-lock` BucketClass and `-readonly` BucketAccessClass were dropped (breaking access control), the S3 service DNS name regressed to a non-prefixed form (breaking in-cluster clients), and `volumeSizeLimitMB` prevented multiple collections from sharing a volume ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2943).

* **fix(backups): carry dropdown option sources in CRD annotations, not schema**: The four `backups.cozystack.io` CRDs embedded `x-cozystack-options` vendor extensions directly in their OpenAPI validation schema. Kubernetes only preserves `x-kubernetes-*` vendor extensions; every other `x-` key was rejected by helm-controller's SSA apply, causing the backup CRDs to never apply and the controller to crash-loop. Option sources are now stored in CRD object annotations, which are preserved unchanged ([**@lexfrei**](https://github.com/lexfrei) in #2823).

* **fix(ingress, platform): deliver `publishing.proxyProtocol` to host ingress-nginx**: The three nginx config keys (`use-proxy-protocol`, `real-ip-header`, `enable-real-ip`) were written into a component values block that had no `install:` section, so the package controller never built a HelmRelease for it and the values were silently dropped. The switch now routes through `_cluster.proxy-protocol` in the `cozystack-values` Secret — the same path `expose-external-ips` already uses to reach the host ingress release ([**@lexfrei**](https://github.com/lexfrei) in #2799).

* **fix(platform): add cert-manager dependency to webhook-cert consumers**: Five PackageSources (`capi-operator`, `cozystack-engine`, `ingress-nginx`, `kamaji`, `rabbitmq-operator`) now declare `dependsOn: cert-manager`. Without this, on cold installs the chart applied `cert-manager.io/v1 Certificate` CRs before the cert-manager webhook was ready, burning 2–3 minutes per affected package in retry loops ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2726).

* **fix(info): use root-host for Keycloak OIDC issuer URL in tenant kubeconfig**: Non-root tenants had `--oidc-issuer-url` pointing at `keycloak.<TENANT>.<ROOT-DOMAIN>`, but Keycloak lives at `keycloak.<ROOT-DOMAIN>`. `kubectl oidc-login` failed TLS verification against the nginx default cert, making the auto-generated kubeconfig unusable for any non-root tenant ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2704).

* **fix(kubernetes): add `config_path` patch for containerd 2.x**: Adds support for containerd 2.x where the CRI plugin section was renamed from `io.containerd.grpc.v1.cri` to `io.containerd.cri.v1.images`. The patch regexes are also made resilient to single and double quotes around plugin names, fixing the issue on Ubuntu 24.04 where containerd 2.x uses single quotes ([**@elaugaste**](https://github.com/elaugaste) in #2723).

* **fix: add `startupProbe` to `kubevirt-cdi-operator` and `grafana-operator`**: Both operators were missing a `startupProbe`, causing them to be killed by the `livenessProbe` during slow startup under load (slow leader election for `kubevirt-cdi-operator`, startup latency under high cluster activity for `grafana-operator`) ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2725).

* **fix(kubernetes): add `spec.timeout` to tenant CSI HelmRelease**: The `kubevirt-csi-node` HelmRelease template was the only tenant addon HR missing `spec.timeout: 10m`, causing it to use Flux's default (5 minutes), which was insufficient on loaded clusters. All other tenant addon HelmReleases (cilium, coredns, ingress-nginx, etc.) already set 10m ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2727).

* **fix(dashboard): unblock token-proxy startup when JWKS is briefly unreachable**: The dashboard's token-proxy process failed to start if the Keycloak JWKS endpoint was briefly unreachable during pod startup. This caused the dashboard to stay in a crash loop until Keycloak became available, rather than retrying gracefully ([**@lexfrei**](https://github.com/lexfrei) in #2745).

* **fix(prometheus-operator-crds): ship full upstream CRD bundle**: A `rm -f` line in the Makefile stripped 6 of the 10 upstream CRDs (`Alertmanager`, `AlertmanagerConfig`, `Prometheus`, `PrometheusAgent`, `ScrapeConfig`, `ThanosRuler`). Operators depending on any of these CRDs were silently missing them from their clusters ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2660).

* **fix(capi): add `startupProbe` to `capi-controller-manager`**: The CAPI controller manager lacked a startupProbe, causing it to be killed by the liveness probe during slow startup (e.g. when cert provisioning was delayed). This manifested as repeated restarts in clusters with high cert-manager load ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2946).

* **fix(kafka): reclaim ZooKeeper PVCs on Kafka release deletion** *(backported to v1.4.1)*: Deleting a Kafka application freed broker PVCs but orphaned ZooKeeper PVCs, requiring manual cleanup before the same release name could be reused. `deleteClaim: true` is now set on ZooKeeper storage ([**@Arsolitt**](https://github.com/Arsolitt) in #2679).

* **fix(postgres): accept integer values for `postgresql.parameters` in schema** *(backported to v1.4.1)*: `postgresql.parameters` declared `{type: string}` only, rejecting bare integers like `max_connections: 100`. An `intOrString` type alias is now emitted with `anyOf: [integer, string]` plus `x-kubernetes-int-or-string: true`. The same PR extends the `$dangerousParams` blocklist with `archive_cleanup_command` and `recovery_end_command` ([**@IvanHunters**](https://github.com/IvanHunters) in #2687).

* **fix(platform): migrate `ephemeralStorage` to `diskSize` via pre-upgrade hook** *(backported to v1.4.1)*: PR #2454 renamed `nodeGroups[*].ephemeralStorage` to `nodeGroups[*].diskSize` with a hard `{{ fail }}` guard. A new platform pre-upgrade migration (migration 41) now walks all `kuberneteses.apps.cozystack.io` Application CRs and renames the field automatically before chart resources are applied ([**@IvanHunters**](https://github.com/IvanHunters) in #2688).

* **fix(cluster-api): fix Kamaji OOM and set limits on unset providers** *(backported to v1.4.1)*: The Kamaji control-plane-provider resource override targeted a container named `manager`, but the image names it `controller` — the intended 1 CPU / 1024 Mi limits were silently dropped and Kamaji used 128 Mi, causing repeated OOMKilled restarts. The container name is corrected and modest resource requests and limits are added for the core, kubeadm-bootstrap, and kubevirt-infrastructure providers that previously ran as BestEffort ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2708).

* **fix(opensearch-operator): replace deprecated `kube-rbac-proxy` image** *(backported to v1.4.1)*: The `gcr.io/kubebuilder/kube-rbac-proxy` image is no longer available. Replaced with `quay.io/brancz/kube-rbac-proxy` following the convention already used by etcd-operator and other components ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2689).

* **fix(platform): add OpenSearch to PaaS bundle** *(backported to v1.4.2)*: The `opensearch-operator` and `opensearch-rd` PackageSources were missing from the PaaS bundle template, so the OpenSearch operator and its ApplicationDefinition were never deployed — tenants could not provision OpenSearch instances ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2648).

* **fix(csi): route RWX Block volumes to upstream hotplug detach** *(backported to v1.4.2)*: `ControllerUnpublishVolume` selected the NFS cleanup branch using only `hasRWXAccessMode(pvc)`, so RWX Block PVCs (KubeVirt live-migration disks) were sent down the NFS path and left their `DataVolume` entry in `VM.spec.template.spec.volumes` permanently, blocking future attaches. The fix introduces `isNFSVolume(pvc)` requiring both `ReadWriteMany` and `VolumeMode=Filesystem` ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2658).

* **fix(csi): verify VMI Ready after `kubevirt-csi` Publish to surface stuck-PVC failures** *(backported to v1.4.2)*: Upstream `ControllerPublishVolume` reported success as soon as the volume entry appeared in `VM.spec.template.spec.volumes`, even if QEMU never received `device_add`. The wrapper now re-reads `VMI.Status.VolumeStatus` and returns `codes.Unavailable` if the volume is not in `VolumeReady` phase, keeping `external-attacher` retrying and surfacing the underlying failure at the CSI layer ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2659).

* **fix(dashboard): grant tenant dashboard read on `cozy-public` PVCs** *(backported to v1.4.3)*: The VM disk source-image dropdown returned 403 and stayed empty even when golden images existed in the `cozy-public` namespace. `get/list/watch` on PersistentVolumeClaims is now added to the `cozy:tenant:dashboard` Role ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2843).

* **fix(api): emit `initial-events-end` bookmark for `core.cozystack.io` watches** *(backported to v1.4.3)*: `TenantSecret`, `TenantModule`, and `TenantNamespace` aggregated API resources never sent the `k8s.io/initial-events-end` bookmark required by the WatchList protocol. Client-go informers using `WatchListClient` never reached `HasSynced` and logged warnings every ~10 seconds ([**@sunib**](https://github.com/sunib) in #2786).

* **fix(networking): point host ouroboros proxy at the root-tenant ingress** *(backported to v1.4.3)*: When `publishing.proxyProtocol` was enabled, the host-level ouroboros proxy inherited the wrong backend FQDN (a managed Kubernetes tenant cluster address). On the host, ingress-nginx is deployed as `root-ingress-controller` in `tenant-root`, so the proxy never became `Ready` ([**@lexfrei**](https://github.com/lexfrei) in #2800).

* **fix(objectstorage-controller): propagate Bucket readiness to BucketClaim** *(backported to v1.4.3)*: The vendored COSI controller hardcoded `bucketReady=false` after dynamic provisioning and never re-read the `Bucket` to pick up the `true` transition, so `BucketAccess` was never granted and buckets ended up without credentials ([**@lexfrei**](https://github.com/lexfrei) in #2792).

* **fix(kubernetes): stamp application lineage labels on worker node VMs** *(backported to v1.4.3)*: Worker-node VMs created by Cluster API were never stamped with `apps.cozystack.io/application.{group,kind,name}` lineage labels, preventing the dashboard from attributing their virt-launcher pods to the owning Kubernetes application. A companion fix quotes `application.name` so a purely-numeric cluster name renders as a YAML string ([**@kvaps**](https://github.com/kvaps) in #2779).

## Security

* **ci: add CodeQL workflow for Go (SAST)**: Adds a weekly CodeQL static analysis scan for the Go codebase ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2851).

* **ci: add OpenSSF Scorecard workflow**: Enables continuous OpenSSF Scorecard scoring for the repository, covering vulnerability checks, branch protection, CI best practices, and contributor risk ([**@tym83**](https://github.com/tym83) in #2720).

* **ci: pin Scorecard workflow actions by SHA**: Pins all third-party Actions used by the Scorecard workflow to digest-pinned SHAs to prevent supply-chain attacks via tag mutation ([**@tym83**](https://github.com/tym83) in #2721).

* **fix(deps): bump Go to 1.26.4 and `x/net` to v0.55.0 to close OSV advisories**: Clears 12 Open Source Vulnerability advisories: 10 against the Go toolchain (fixed in 1.26.3 and 1.26.4) and 2 against `golang.org/x/net` (fixed in v0.55.0). This brings the OpenSSF Scorecard Vulnerabilities check to a passing state ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2852).

## Dependencies

* **chore(seaweedfs): bump `seaweedfs-cosi-driver` to v0.3.1** *(backported to v1.4.3)*: v0.3.1 ships a stale-socket self-heal: the COSI driver now removes any leftover UNIX socket before binding, so the objectstorage provisioner recovers automatically from a non-graceful exit instead of wedging on "bind: address already in use" ([**@lexfrei**](https://github.com/lexfrei) in #2791).

## Development, Testing, and CI/CD

* **feat(ci): test-impact analysis for E2E**: A new pre-E2E step analyses which packages changed in the PR diff and selects only the relevant bats test files. E2E runs that previously exercised the full test matrix now run only the impacted tests on normal PRs, with full-suite runs reserved for release PRs and `[full-e2e]` labelled PRs ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2559).

* **ci(workflows): add SSH breakpoint on E2E failure for debug PRs**: E2E jobs on PRs labelled `debug` or `release` now drop into an interactive SSH session on failure, so engineers can inspect the cluster state without needing to reproduce the failure locally ([**@kvaps**](https://github.com/kvaps) in #2535).

* **ci(release): auto-patch-release only the 2 newest release lines**: The automated patch-release cron now restricts backport CI to the two most recent `release-X.Y` branches, preventing unnecessary runs on end-of-life lines ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2856).

* **ci(release): repair orphaned draft release on retag** *(backported to v1.4.3)*: When a git tag was deleted and re-created between draft creation and merge, GitHub orphaned the draft release by setting `tag_name` to `"untagged-<hash>"`. The finalize step now detects the orphaned form, falls back to matching by `name`, repairs `tag_name` via `updateRelease`, and then publishes ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2761).

* **refactor(build): standardize image tagging to fix concurrent PR push conflicts**: Aligns image tagging across all build targets to prevent tag collision when multiple PRs push to the same registry prefix concurrently ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2711).

* **refactor(build): mode=max registry cache with a main-only warmer**: Switches Docker BuildKit cache to `mode=max` and introduces a dedicated cache-warmer job that runs only on `main` pushes. PR builds warm-start from the registry cache without the warmer running on every PR ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2938).

* **ci(build): isolate each PR build on its own ephemeral runner VM**: PR builds are now dispatched to fresh ephemeral runner VMs, preventing state bleed between concurrent builds and improving security isolation ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2939).

* **test(api): add read-path authorization tests for `TenantNamespace` registry**: Adds direct unit coverage for the authorization-sensitive read paths (`Get`, `List`, `Watch`) of the TenantNamespace registry, including per-user filtering, privileged-group bypass, bookmark forwarding, field and label selector filtering, and goroutine lifecycle. A production gap (Watch goroutine never closing its result channel) is fixed as part of this work ([**@lexfrei**](https://github.com/lexfrei) in #2864).

* **ci: add Go-scoped Renovate configuration**: Enables automated dependency updates for Go modules in the repository via Renovate ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2850).
