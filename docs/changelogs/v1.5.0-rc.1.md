# Cozystack v1.5.0-rc.1

Cozystack v1.5.0-rc.1 is the first release candidate for v1.5.0, featuring **Cilium Gateway API** as the new ingress layer for tenant and system services, a new namespaced **Option resource** for live-cluster-sourced dynamic form dropdowns in the dashboard, **TLS support** added to Kafka, NATS, Qdrant, and PostgreSQL managed apps, a generic **Job backup strategy** for any application type, a **deletion-protection guardrail** via ValidatingAdmissionPolicy, and a **SeaweedFS split** into dedicated database and storage-system tiers. Additional highlights include GPU passthrough enabled out-of-the-box for tenant Kubernetes, VM start/stop/restart RBAC for tenants, Flux v2.8.0 upgrade with strict SSA and kstatus, **Kuberture** as an optional system package for publishing the Kubernetes API endpoint via DNS, and StorageClass immutability for stateful apps.

> **Note:** Fixes marked with *(backported to v1.4.x)* were also included in v1.4.1 or v1.4.2 patch releases.

## Feature Highlights

### Cilium Gateway API: Native Ingress for Tenant Services

The platform ingress layer is now powered by **Cilium Gateway API**, replacing ingress-nginx as the default for tenant and system services. A new `TenantGateway` CRD is reconciled by `cozystack-controller`: a tenant opts in with `tenant.spec.gateway=true`, which causes the controller to materialise a `Gateway`, per-tenant cert-manager `Issuer`, `Certificate` (wildcard or per-listener), and HTTP→HTTPS redirect `HTTPRoute`. Tenants that do not own a Gateway inherit through the parent tree via the `namespace.cozystack.io/gateway` label — same propagation shape as the existing ingress inheritance.

Two certificate modes are supported: **HTTP-01** (the default, zero platform-side config, per-app listener certificate) and **DNS-01** (wildcard certificate with SAN expansion for child tenant apex domains, four DNS providers supported). The Gateway's `allowedRoutes.namespaces.selector` is label-keyed, so system namespaces and inheriting child tenants are admitted automatically. Defaults remain legacy (`gateway.enabled=false`), so existing clusters upgrade unchanged — Gateway API is strictly opt-in ([**@lexfrei**](https://github.com/lexfrei) in #2470).

### Option Resource: Live-Cluster-Sourced Dynamic Dropdowns

A new namespaced `Option` resource (`core.cozystack.io/v1alpha1`, read-only: list/get/watch) enables dashboard form fields — GPU device names, KubeVirt instancetypes/preferences, Multus networks, VM images, bucket storage pools, storage classes, and backup class/plan/job fields — to render as **dropdowns populated from live cluster state** rather than free-text or static enums.

Items are computed on read by an in-process provider registry using a privileged apiserver client, so tenants receive curated option lists without needing broad cluster-read permissions. Option providers implemented in this release: `gpu`, `instancetype`, `instanceprofile`, `network`, `image`, `storagepool`, `storageclass`, `backupclass`, `plan`, `backup`, and `appkind`. App `values.yaml` files declare the new `x-cozystack-options` schema keyword (via the `@x-cozystack-options` cozyvalues-gen directive), which is preserved through schema generation into served OpenAPI and consumed by the dashboard to render the annotated field as a dropdown. Tenants receive read access to `options` in their namespace via the aggregated `cozy:tenant:view:base` ClusterRole ([**@kvaps**](https://github.com/kvaps) in #2778).

### TLS Support for Managed Apps

TLS is now available as an opt-in feature for four managed application types, using a consistent tri-state `tls.enabled` field (`null` = unset/inherit, `true` = on, `false` = off) and operator-managed cert-manager certificates:

- **Kafka**: TLS via Strimzi listener configuration — a dedicated `tls` listener alongside the existing plaintext listener, cluster CA secret exposed as `<name>-cluster-ca-cert`, and external listener decoupled from TLS state ([**@Arsolitt**](https://github.com/Arsolitt) in #2681).
- **NATS**: TLS via cert-manager — ECDSA P-256 CA and leaf keys, wildcard SAN for cluster routes with `useFQDN=true`, Reloader-triggered pod restart on cert rotation ([**@Arsolitt**](https://github.com/Arsolitt) in #2684).
- **Qdrant**: TLS via cert-manager — ECDSA P-256 keys, per-replica short-form SANs for p2p TLS hostname verification, Reloader annotation for secret rotation ([**@Arsolitt**](https://github.com/Arsolitt) in #2685).
- **PostgreSQL**: TLS via CNPG operator-managed certificates with extra-SAN injection; CA retrieved by tenants from the user-credentials Secret rather than a separate CA Secret ([**@Arsolitt**](https://github.com/Arsolitt) in #2686).

### Job Backup Strategy: Generic Application-Agnostic Backups

The `strategy.backups.cozystack.io/Job` backup strategy enables **any application** to define custom backup and restore logic via a user-supplied Kubernetes `batch/v1.Job` template. Strategy parameters are resolved from a `BackupClass` into a `ResolvedBackupConfig` and passed as template context: `.Application` (the live application object), `.Release.Name`, `.Release.Namespace`, `.Mode` (`backup` or `restore`), `.Parameters`, and `.Backup` (restore runs only). Parameters round-trip through the `Backup`'s `driverMetadata`, so a restore re-renders against the same values used during backup. The implementation follows the same lifecycle as the existing per-driver strategies (render `PodTemplateSpec` → run a one-shot `Job` → poll for terminal condition → emit a `Backup` artifact) and reuses all shared controller helpers. No additional RBAC is required ([**@lllamnyp**](https://github.com/lllamnyp) in #1721).

### Deletion-Protection Guardrail via ValidatingAdmissionPolicy

Critical platform objects — including the `cozy-system` Namespace, `tenant-root` Namespace and HelmRelease, `cozystack-version` ConfigMap, `cozystack-packages` OCIRepository, cert-manager ClusterIssuers, the LinstorCluster, and the `packages.cozystack.io`/`packagesources.cozystack.io` CRDs — are now protected from accidental deletion by a **ValidatingAdmissionPolicy** that rejects `DELETE` requests on objects labeled `platform.cozystack.io/no-delete=true`. The VAP is evaluated in-process by the Kubernetes apiserver (no webhook, no DaemonSet, no TLS) and is gated on a Kubernetes 1.30+ capability check. The VAP and its binding self-protect by carrying the same label.

Migration 42 backfills the `no-delete` label on the `cozystack-version` ConfigMap on upgrade. For planned teardown, the `make unprotect` target and `unprotect.sh` script strip the label. `NOTES.txt` warns when the VAP is silently skipped on clusters below the 1.30 threshold ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2650).

### SeaweedFS Split: Dedicated Database and Storage-System Tiers

The monolithic `seaweedfs-system` HelmRelease is split into two independent releases: **`seaweedfs-db`** (the CNPG `Cluster` backing SeaweedFS volume metadata) and **`seaweedfs-system`** (the filer and S3 services), with `seaweedfs-system` declaring a `dependsOn` relationship to `seaweedfs-db`. The new `seaweedfs-db` release uses Flux v2.8.0 `healthCheckExprs` with a CEL expression on `Cluster.status.conditions[type=Ready]`, so its `Ready=True` flips only after the postgres primary is actually serving connections — eliminating the filer CrashLoopBackOff race on fresh tenant installs caused by socket-LB returning `EPERM` to ClusterIPs with no ready endpoints.

The split also enables independent resource tuning for each tier via `seaweedfs-db.spec.values`. Migration 43 adopts existing `Cluster/seaweedfs-db` resources into the new release on upgrade. The parent HelmRelease timeout is bumped from 5 m to 15 m to cover the serialized `seaweedfs-db → seaweedfs-system → seaweedfs` chain ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2601).

## Major Features and Improvements

* **[platform] Add Kuberture as an optional system package**: The new opt-in `cozystack.kuberture` package deploys [Kuberture](https://github.com/lexfrei/kuberture), a small controller that translates the `default/kubernetes` EndpointSlice into annotated headless Services, bridging the gap that external-dns cannot consume EndpointSlice sources directly. Operators populate `config.outputs` per HelmRelease — each output targets a specific external-dns instance by its `annotationPrefix`, so a single Kuberture install can serve multiple external-dns instances simultaneously. Enable via `bundles.enabledPackages: [cozystack.kuberture]` ([**@lexfrei**](https://github.com/lexfrei) in #2647).

* **[platform] Add default BackupClass**: The platform now ships a default `BackupClass` resource that tenants can reference without requiring per-cluster configuration for common backup scenarios. This closes the last gap for out-of-the-box backup capability for managed applications ([**@androndo**](https://github.com/androndo) in #2716).

* **[platform] Add etcd backup strategy controller**: The `backupstrategy-controller` is extended with a new **etcd** strategy that handles `BackupJob` (snapshot → `Backup` artifact) and destructive in-place `RestoreJob` flows for etcd clusters. Snapshot metadata is persisted in the `Backup` object, and the destination is validated before restore. S3-only storage is supported in this initial implementation ([**@androndo**](https://github.com/androndo) in #2641).

* **[flux] Upgrade to Flux v2.8.0 with strict SSA and kstatus**: Bumps `flux-operator` and `flux-instance` charts to v0.50.0 (Flux v2.8.0), enabling strict SSA (`forceApply`) and kstatus-aware reconciliation for more reliable HelmRelease health checks. Platform-wide `upgrade.force: true` flags are removed (Helm v4 in helm-controller v1.5.0 rejects combining `--force-replace` with SSA's `--force-conflicts`). Folds the earlier `flux-operator/flux-instance v0.48.0` opt-in from the same series ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2602).

* **[kubernetes] Enable GPU passthrough out-of-the-box**: Tenant Kubernetes worker nodes in GPU node groups now default `NVreg_NvLinkDisable=1` (prevents NVLink peer mesh initialization that fails with passthrough GPUs) and are labeled `gpu=on`. This enables GPU-capable workloads to schedule automatically without per-cluster operator tuning ([**@kvaps**](https://github.com/kvaps) in #2780).

* **[rbac] Allow tenants to start/stop/restart VMs**: Tenant service accounts can now issue `start`, `stop`, and `restart` subresource calls on `VirtualMachine` objects in their namespace, enabling automated VM lifecycle management from within tenant workloads or CI pipelines ([**@kvaps**](https://github.com/kvaps) in #2777).

* **[dashboard] Add cluster-usage RBAC for the admin page**: Extends the dashboard RBAC bundle with the read permissions required by the new cluster-usage admin page, allowing operators to view aggregate resource consumption across tenants without additional ClusterRole bindings ([**@lexfrei**](https://github.com/lexfrei) in #2743).

* **[apps] Mark stateful-app storageClass fields as immutable**: The `storageClass` field in stateful application `values.yaml` files is now annotated as immutable in the schema. Attempts to change `storageClass` on an existing stateful app are rejected at the API layer, preventing data-loss scenarios that would result from silent storage reclasses ([**@lexfrei**](https://github.com/lexfrei) in #2639).

## Bug Fixes

* **[kubernetes] Stamp application lineage labels on worker node VMs**: Worker node VirtualMachine objects are now stamped with `app.kubernetes.io/managed-by`, `app.kubernetes.io/instance`, and related lineage labels, enabling cluster-wide tooling to correlate VMs with their owning Kubernetes application and quota scope *(backported to v1.4.2)* ([**@kvaps**](https://github.com/kvaps) in #2779).

* **[dashboard] Unblock token-proxy startup when JWKS endpoint is briefly unreachable**: The token-proxy container no longer fails its startup check when the JWKS endpoint is temporarily unavailable during platform bootstrap (e.g. Keycloak not yet ready). The proxy now retries with backoff instead of crashing, eliminating a race condition that caused dashboard pods to restart-loop during initial cluster setup *(backported to v1.4.2)* ([**@lexfrei**](https://github.com/lexfrei) in #2745).

* **[platform] Add cert-manager dependency to webhook-cert consumers**: HelmReleases that consume cert-manager-managed certificates (cozystack-engine OIDC variant and webhook-cert consumers) now declare an explicit `dependsOn: cert-manager` in their HelmRelease spec, preventing reconcile failures when cert-manager is not yet ready ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2726).

* **[platform] Add startupProbes for kubevirt-cdi-operator and grafana-operator**: Adds `startupProbe` definitions to both operators to give them sufficient time for leader election and slow startup under load, preventing premature liveness-probe failures and OOMKill-recovery crashes from being misclassified as crash loops ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2725).

* **[info] Use root-host for Keycloak OIDC issuer URL in tenant kubeconfig**: The Keycloak OIDC issuer URL written into tenant kubeconfig was using the per-tenant host instead of the root platform host, causing `oidc-login` to fail for tenants whose Keycloak realm lives on the root domain ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2704).

* **[csi] Verify VMI Ready after kubevirt-csi Publish to surface stuck-PVC failures**: Upstream `ControllerPublishVolume` has a fast path that reports success as soon as a volume entry appears in `VM.spec.template.spec.volumes`, even if the hotplug never completed due to the backing PVC being stuck `ClaimPending`. The wrapper now re-reads `VMI.Status.VolumeStatus` and returns `codes.Unavailable` if the volume is not in `VolumeReady` phase, keeping `external-attacher` retrying and surfacing the underlying provisioning failure at the CSI layer rather than as an opaque kubelet-side error *(backported to v1.4.2)* ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2659).

* **[csi] Route RWX Block volumes to upstream hotplug detach**: `ControllerUnpublishVolume` was using only `hasRWXAccessMode(pvc)` to detect NFS volumes, incorrectly routing KubeVirt live-migration RWX Block PVCs down the NFS cleanup path (which only deletes a `CiliumNetworkPolicy`). The fix introduces `isNFSVolume(pvc)`, requiring both `ReadWriteMany` access mode **and** `VolumeMode=Filesystem`, so block volumes are correctly forwarded to upstream's hotplug detach path *(backported to v1.4.2)* ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2658).

* **[postgres] Accept integer values for `postgresql.parameters` in schema**: The JSON schema for `postgresql.parameters.additionalProperties` accepted only strings, causing Helm validation failures when users supplied bare integers (e.g. `max_connections: 100`) in Application CR specs. The schema now uses `anyOf: [integer, string]` plus `x-kubernetes-int-or-string: true`. The `$dangerousParams` blocklist is also extended with `archive_cleanup_command` and `recovery_end_command` *(backported to v1.4.1)* ([**@IvanHunters**](https://github.com/IvanHunters) in #2687).

* **[platform] Migrate `ephemeralStorage` to `diskSize` via pre-upgrade hook**: PR #2454 renamed `nodeGroups[*].ephemeralStorage` to `nodeGroups[*].diskSize` and added a hard `{{ fail }}` guard that blocked all HelmRelease reconciliation for clusters still carrying the old field name. Migration 41 (a platform pre-upgrade hook) now walks all `kuberneteses.apps.cozystack.io` Application CRs and renames the field automatically before chart resources are applied *(backported to v1.4.1)* ([**@IvanHunters**](https://github.com/IvanHunters) in #2688).

* **[cluster-api] Fix Kamaji OOM and set resource limits on unset providers**: The Kamaji control-plane-provider resource override targeted a container named `manager`, but the upstream image names it `controller`, silently dropping the intended 1 CPU / 1024 Mi limits and leaving it running with the upstream default of 128 Mi, causing repeated OOMKill restarts. The container name is corrected. Modest requests and limits are also set for the core, kubeadm-bootstrap, and kubevirt-infrastructure providers, which previously ran as BestEffort and were first to be evicted under memory pressure *(backported to v1.4.1)* ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2708).

* **[kafka] Remove ZooKeeper PVCs on Kafka release deletion**: The Strimzi `Kafka` CR set `deleteClaim: true` for broker JBOD volumes but left ZooKeeper persistent-claim storage at `deleteClaim: false`, orphaning ZooKeeper PVCs in the tenant namespace on release deletion and blocking reuse of the same release name. `deleteClaim: true` is now set on the ZooKeeper storage as well *(backported to v1.4.1)* ([**@Arsolitt**](https://github.com/Arsolitt) in #2679).

* **[opensearch-operator] Replace deprecated `kube-rbac-proxy` image**: The `gcr.io/kubebuilder/kube-rbac-proxy` image is no longer available since the kubebuilder GCR registry was sunset. The image is replaced with `quay.io/brancz/kube-rbac-proxy` via a `values.yaml` override, matching the convention used by other components in Cozystack *(backported to v1.4.1)* ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2689).

* **[platform] Add OpenSearch to PaaS bundle**: The `bundles.paas.enabled=true` template was missing includes for `cozystack.opensearch-operator` and `cozystack.opensearch-rd`, so the dashboard catalog showed no OpenSearch entry and tenants could not provision OpenSearch instances despite the packages being installed. The two missing PackageSources are now added *(backported to v1.4.2)* ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2648).

* **[kubernetes] Add `config_path` patch for containerd 2.x**: Containerd 2.x moved its config file from `/etc/containerd/config.toml` to a new location; the `config_path` patch now handles both plugin schema versions, ensuring the Kubernetes containerd configuration is applied correctly on clusters running containerd 2.x ([**@elaugaste**](https://github.com/elaugaste) in #2723).

* **[ci] Include `data/versions` in website docs commit**: The automated website docs commit now includes the `data/versions/*.yaml` files alongside the managed-app reference pages, ensuring version compatibility matrices are always published in sync with a release ([**@myasnikovdaniil**](https://github.com/myasnikovdaniil) in #2707).

* **[controller] Remove dead dashboard controller**: The legacy `dashboard.cozystack.io` CRD and its reconciler are removed — the controller code, CRD manifests, and migration 40 that drops the CRD are cleaned up ([**@kvaps**](https://github.com/kvaps) in #2694).

* **[img] Refresh Cozystack logo and add icon/stacked variants**: Updates the repository logo assets with the new brand mark and adds icon-only and stacked variants for use in badges, favicons, and the website ([**@kvapsova**](https://github.com/kvapsova) in #2746).
