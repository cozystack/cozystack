# Split `seaweedfs-system` HelmRelease into `seaweedfs-db` + `seaweedfs-system`

## Problem

On a fresh tenant install, `seaweedfs-filer` pods CrashLoopBackOff for 60+ seconds with:

```
dial tcp <ClusterIP>:5432 (seaweedfs-db-rw): connect: operation not permitted
```

Cilium runs with `kubeProxyReplacement: true` (socket-LB). When a Service has no Ready endpoints, socket-LB returns `EPERM` from `connect(2)` rather than `ECONNREFUSED`. The seaweedfs umbrella chart deploys the CNPG `Cluster/seaweedfs-db` and the filer StatefulSet in the same Helm release, so the filer pod schedules concurrently with the postgres bootstrap (initdb Job + primary pod, ~55–70 s). Filer's `connect()` returns `EPERM`, the pod exits, kubelet applies exponential backoff (10 s → 20 s → 40 s → …). By the time postgres is Ready, backoff has already pushed the next attempt past the e2e bats `kubectl wait hr/seaweedfs-system --timeout=2m` window, blocking the `Configure Tenant and wait for applications` test in `hack/e2e-install-cozystack.bats`.

The same race affects monitoring (grafana, alerta) but is out of scope for this spec — it gets a follow-up PR using the same pattern.

## Goal

Make `seaweedfs-filer` start only after `Cluster/seaweedfs-db` is `Ready`, eliminating the EPERM crashloop window. Mechanism: split the Helm release in two so Flux's `dependsOn` orders the deploys, and add a Helm post-install hook that gates the database release's `Ready` on the actual CNPG `Cluster` `Ready` condition (not just on `helm install` succeeding).

Non-goals:
- Fixing the same race in monitoring, harbor, openbao, etc. (follow-up issues).
- Changing the user-visible chart values surface (`.Values.db.*` stays in the same place from the user's perspective).

## Architecture

```
Before                              After
──────                              ─────
seaweedfs-system HR                 seaweedfs-db HR  (new)
  ├── Cluster/seaweedfs-db          │   ├── Cluster/seaweedfs-db
  ├── master STS                    │   ├── post-install Hook Job: kubectl wait Cluster Ready
  ├── volume STS                    │   ├── ServiceAccount + Role + RoleBinding for hook Job
  ├── filer STS  ◄── crashloops     │   └── WorkloadMonitor/seaweedfs-db
  ├── s3 Deploy                     │
  ├── cosi Deploy                   seaweedfs-system HR  (existing, slimmed)
  └── WorkloadMonitor/seaweedfs-db    ├── dependsOn: [{ name: seaweedfs-db }]
                                      ├── master STS
                                      ├── volume STS
                                      ├── filer STS  ◄── postgres already Ready
                                      ├── s3 Deploy
                                      └── cosi Deploy
```

Flux's helm-controller does not reconcile a HelmRelease whose `dependsOn` target is not in the `Ready=True` state. The dependent reconcile is requeued at the HR's `interval`. To keep e2e fast we set `seaweedfs-system.spec.interval: 30s` (down from 5 m) so the dependency check happens promptly after `seaweedfs-db` becomes Ready.

The post-install hook Job is the load-bearing piece: without it, the `seaweedfs-db` HR would go Ready as soon as `helm install` succeeded — which happens before the CNPG primary pod is Ready — and `dependsOn` would unblock the app HR too early, reproducing the same race.

## Components

### `packages/system/seaweedfs-db/` — new system package

Mirrors the layout of existing `packages/system/seaweedfs/`:

- `Chart.yaml` — name `cozy-seaweedfs-db`, type `application`
- `values.yaml` — only the `db.*` block lifted out of `packages/system/seaweedfs/values.yaml`
- `values.schema.json` — generated by `make generate`
- `Makefile` — same shape as sibling system packages
- `templates/database.yaml` — moved verbatim from `packages/system/seaweedfs/templates/database.yaml`
- `templates/wait-cluster-ready.yaml` — new Helm post-install/post-upgrade hook Job (see below)
- `templates/rbac.yaml` — ServiceAccount + Role + RoleBinding scoped to the release namespace, granting `get`/`list`/`watch` on `clusters.postgresql.cnpg.io`

The hook Job:

```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: {{ .Release.Name }}-wait-cluster-ready
  annotations:
    helm.sh/hook: post-install,post-upgrade
    helm.sh/hook-weight: "1"
    helm.sh/hook-delete-policy: hook-succeeded,before-hook-creation
spec:
  backoffLimit: 0
  template:
    spec:
      serviceAccountName: {{ .Release.Name }}-wait
      restartPolicy: Never
      containers:
      - name: wait
        image: <small kubectl image, pinned via cozystack image conventions>
        command: ["/bin/sh", "-ec"]
        args:
        - |
          # Tight loop with short timeout: surfaces real failures fast,
          # tolerates the 30–70 s CNPG bootstrap window.
          until kubectl wait cluster.postgresql.cnpg.io/seaweedfs-db \
            -n {{ .Release.Namespace }} \
            --for=condition=Ready --timeout=15s; do
            echo "waiting for seaweedfs-db Cluster Ready…"
            sleep 5
          done
```

The HR's overall `timeout: 10m` (already configured) is the outer ceiling; if CNPG is genuinely broken, the Job's outer timeout from Helm causes the HR to fail-with-retry, surfacing the breakage instead of hiding it.

### `packages/extra/seaweedfs/templates/seaweedfs-db.yaml` — new wrapper template

Emits `HelmRelease/{{ .Release.Name }}-db` referencing `cozystack-seaweedfs-application-default-seaweedfs-db` (the new ExternalArtifact). Passes only `.Values.db.*`. Keeps the existing labels (`sharding.fluxcd.io/key: tenants`, etc.). `interval: 5m`, `timeout: 10m`. The WorkloadMonitor for the db (currently in `seaweedfs.yaml`) moves here.

### `packages/extra/seaweedfs/templates/seaweedfs.yaml` — modified

- Add `spec.dependsOn: [{ name: "{{ .Release.Name }}-db" }]` to the existing `*-system` HR.
- Lower `spec.interval` from `5m` to `30s` so the dependency check fires promptly after `seaweedfs-db` flips Ready.
- Remove the WorkloadMonitor for `seaweedfs-db` (moved to the new template).
- Stop passing `.Values.db.*` to the system HR's `values:` block (it no longer needs them — the Cluster lives in the db HR).

### `packages/system/seaweedfs/` — modified

- Delete `templates/database.yaml`.
- Remove `db.*` from `values.yaml`.
- Re-run `make generate` to regenerate `values.schema.json` and `README.md`.

### Source registration

`packages/core/platform/sources/seaweedfs-application.yaml` (or wherever the existing seaweedfs-application chart is registered) gets a sibling entry that builds and exposes the new `seaweedfs-db` system chart as `cozystack-seaweedfs-application-default-seaweedfs-db`. The exact mechanism follows the project's two-step system-pkg registration (verified by reading how a recent system-pkg addition was wired up — TBD during planning).

### Migration

`packages/core/platform/images/migrations/migrations/<next-number>` — POSIX shell script (matches the existing migrations/31 style):

```sh
#!/bin/sh
set -euo pipefail

# For every namespace that currently has a Cluster/seaweedfs-db owned by
# the seaweedfs-system Helm release, rewrite the helm ownership annotations
# so the new seaweedfs-db release adopts it.
for ns in $(kubectl get cluster.postgresql.cnpg.io -A -o jsonpath='{range .items[?(@.metadata.name=="seaweedfs-db")]}{.metadata.namespace}{"\n"}{end}'); do
  current=$(kubectl get cluster.postgresql.cnpg.io seaweedfs-db -n "$ns" \
    -o jsonpath='{.metadata.annotations.meta\.helm\.sh/release-name}' 2>/dev/null || true)
  if [ "$current" = "seaweedfs-system" ]; then
    echo "Re-annotating Cluster/seaweedfs-db in $ns"
    kubectl annotate cluster.postgresql.cnpg.io seaweedfs-db -n "$ns" \
      meta.helm.sh/release-name=seaweedfs-db \
      helm.sh/resource-policy=keep \
      --overwrite
    # release-namespace stays the same (tenant namespace, e.g. tenant-root)
  fi
done

# Stamp version
kubectl create configmap -n cozy-system cozystack-version \
  --from-literal=version=<next> --dry-run=client -o yaml | kubectl apply -f -
```

Idempotent (the `if` guards against re-runs). Runs before the upgraded operator reconciles the new HR, so the new `seaweedfs-db` chart adopts an existing Cluster instead of creating a duplicate.

The Cluster CR also needs `helm.sh/resource-policy: keep` set on it as a defense-in-depth — added once via the migration so even if the seaweedfs-system upgrade somehow runs first, it cannot delete the Cluster.

## Data flow

Fresh install of a tenant with `seaweedfs: true`:

```
1. Tenant CR patched           → creates HR `seaweedfs` (in extra layer)
2. HR `seaweedfs` reconciles   → creates HRs `seaweedfs-db` + `seaweedfs-system`
                                  (both in tenant namespace)
3. helm-controller picks up:
   - `seaweedfs-db`           → reconciles immediately
   - `seaweedfs-system`       → blocked, dependsOn seaweedfs-db
4. `seaweedfs-db` install:
   - applies Cluster CR
   - hook Job runs: polls Cluster Ready (kubectl wait, 5 s sleep loop)
   - CNPG operator: provisions PVC, runs initdb Job, starts primary pod
   - primary pod becomes Ready (~30–60 s)
   - hook Job's `kubectl wait` returns 0 → Job Succeeded
   - helm install marks Ready → HR Ready=True
5. `seaweedfs-system` reconciles within 30 s (interval), passes dependsOn check
   - applies master/volume/filer/s3/cosi
   - filer pod schedules, connects to seaweedfs-db-rw — endpoints exist, no EPERM
   - HR Ready=True
```

Upgrade path: same as above, except the migration script runs first (re-annotates the existing Cluster). The new `seaweedfs-db` HR adopts the existing Cluster (Helm sees matching annotations, treats as upgrade not install). Hook Job runs on `post-upgrade` too, so Ready gating still applies.

## Error handling

- **Hook Job fails** (Cluster never becomes Ready, e.g. PVC stuck): Helm install times out at the HR's `timeout: 10m`, HR enters `Failed` state. Operator/user sees a real error instead of the previous silent CrashLoopBackOff. Existing `install.remediation.retries: -1` re-runs install.
- **Migration runs against a namespace where no Cluster exists**: the `for ns in $(…)` loop yields nothing; script exits 0.
- **Migration runs twice**: idempotent — the `if [ "$current" = "seaweedfs-system" ]` guard skips already-migrated Clusters.
- **Old HR (seaweedfs-system) still owns the Cluster after migration script fails partway**: Helm refuses to manage the Cluster from the new release because annotations don't match. Surfaces as `seaweedfs-db` HR `InstallFailed`. Re-running the migration fixes it.

## Testing

- **Unit (helm-unittest)**: a new test under `packages/system/seaweedfs-db/tests/` asserts the Cluster CR and hook Job render with expected values; existing `packages/system/seaweedfs/tests/` is updated to assert the Cluster is no longer present.
- **E2E (bats)**: no change to `hack/e2e-install-cozystack.bats` itself — it should now pass on a fresh install without the 2 m timeout being hit. Verification = the existing `kubectl wait hr/seaweedfs-system -n tenant-root --timeout=2m --for=condition=ready` line passes consistently.
- **Manual upgrade smoke**: dev cluster on v1.3.x → apply this PR → verify `Cluster/seaweedfs-db` is adopted (annotations rewritten, no data loss, filer reconnects).
